Total é somado no cliente a partir dos preços já públicos do catálogo só para exibição/gate de UX (nunca enviado ao backend nem tratado como fonte de verdade — `checkout_close` sempre recalcula e é quem decide o valor real; se preço mudar entre a montagem e o fechamento, o backend responde e a tela mostra o erro, não um total desatualizado). **Gorjeta não é suportada**: `checkout_close` exige que a soma de `payments` feche exatamente com o subtotal calculado (senão erro `payments do not reconcile with order total`), então não há onde um valor extra de gorjeta iria — item do roteiro (`§6`) que não pôde ser implementado por não existir suporte no schema, registrado aqui em vez de inventado no frontend. Split de pagamento (múltiplas formas de pagamento por comanda) é suportado e testado. `Idempotency-Key` é gerada (`crypto.randomUUID()`) só ao entrar na etapa de pagamento e reaproveitada em reenvios da mesma tentativa, nunca em uma nova tentativa após editar o carrinho (`PWA_PLANEJAMENTO.md §4.3`). Erros de `checkout_close` (`backend/src/shared/rpcError.js` + as mensagens já seguras da migration) são mapeados por substring para PT-BR em `comandaErrors.js` (estoque insuficiente, item indisponível, profissional de pacote incompleto, permissão), com fallback para o texto original em inglês. Papel `professional` não tem nenhuma rota de escrita em `checkout`/leitura em `orders` no backend (nunca esteve nos allowlists dessas rotas, não é uma restrição nova) — o módulo mostra uma mensagem explícita em vez de simular uma comanda que não pode ser fechada. **"Abrir comanda" a partir de um agendamento** (Agenda → detalhe do agendamento → botão novo) navega para `/comanda?appointment_id=...`, que pré-carrega cliente e o serviço do agendamento (profissional já preenchido); não existe vínculo persistido `appointment_id`↔`order` no schema — é só conveniência de UI — então, ao fechar com sucesso, uma segunda chamada independente e best-effort marca o agendamento como `completed` (não bloqueia a comanda já fechada se essa chamada falhar). 18 testes novos (Vitest + Testing Library): 6 de `ComandaPage` (mensagem de indisponibilidade para `professional`, catálogo vazio, gate de fechamento por profissional pendente, fechamento completo com pagamento reconciliado, mapeamento de conflito de estoque, pré-carga por `appointment_id`) e os 3 testes de `AppointmentModal`/Agenda ajustados para o novo `ClientPicker` compartilhado e para o botão "Abrir comanda" (que exigiu envolver os testes de Agenda em `MemoryRouter`, já que o modal agora usa `useNavigate`). Verificado end-to-end no browser real (Supabase + backend locais, org de teste da Fase 6.1/6.2, com fixtures de produto/pacote inseridas via SQL direto — Catálogo/Estoque ainda não têm UI, Fase 6.4): comanda avulsa com serviço + produto + pacote de dois profissionais fechada com split de pagamento (dinheiro + Pix) reconciliado; tentativa de fechar com produto acima do estoque disponível corretamente rejeitada pelo backend com a mensagem em PT-BR; `order_items` conferido no Postgres mostrando a expansão do pacote em uma linha de serviço por componente com o profissional certo e o preço do pacote alocado; fluxo "Abrir comanda" a partir de um agendamento pré-carregando cliente/serviço/profissional e marcando o agendamento como "Concluído" após o fechamento — tudo sem erro de console. `npm test` (32 testes) e `npm run build` seguem passando.
